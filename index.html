1. Authentication Flow Architecture

KeyLine Auth processes standard multi-factor identity challenges via an asynchronous distributed lifecycle. Instead of using complex SDK libraries, your application utilizes browser redirects linked to dynamic database-validated clients.

2. GET /oauth/authorize

Redirect users to this URL to trigger the KeyLine OTP interface screen:

https://keyline-l1n2.onrender.com/oauth/authorize?client_id=YOUR_CLIENT_ID&redirect_uri=YOUR_REDIRECT_URI&response_type=code&scope=openid

Query Parameters Table:

Parameter Type Required Description
client_id String True Your unique hash obtained inside the mobile app ecosystem.
redirect_uri String True Must exactly match one of your application's whitelisted allowedOrigins.
response_type String True Strictly enforce value to code for token delegation.

3. Dynamic CORS Handshake Mechanics

Our servers maintain zero static wildcards (*). When your client triggers a preflight OPTIONS handshake, the system automatically checks the extraction header against database.json list properties. Ensure you declare localhost addresses separately for sandbox routines.

4. POST /oauth/token (Callback Response)

Once authorized, KeyLine calls back your redirect URI with a code parameter. Exchange it via a standard secure server-side POST payload request:

// Sample Backend JSON Request Payload

{

"client_id": "kl_client_5o4myzndm3zonrrr",

"code": "auth_transaction_temp_hash_value",

"grant_type": "authorization_code"

}

5. Error Resolution Directory

CORS_ORIGIN_DENIED:

The domain making the client handshake request does not match the assigned client configuration arrays inside database.json records.

OTP_MISMATCH_STRING:

Triggered when validation parameters fail the sanitized trim string match sequences during active token computation.
